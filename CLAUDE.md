

# Orient（定向）

## 定位

这个阶段的任务是建立项目的第一性原理。

first-principles.md 是项目的最高裁决依据。后续所有开发阶段的每一个技术决策——架构选型、实现方式、取舍判断——都可以通过这份文件自然地推理出来。这是专门用于确立根本原则的阶段，后续开发专注于执行，不再有机会重新审视这些基础问题。在这里遗漏的洞察会成为下游所有阶段的方向性盲区。

Orient 的任务是和用户一起写出这份文件。

## 产出：first-principles.md

这份文件是一个权力结构，不是平铺的列表。上层原则约束下层，冲突时上层否决下层，没有商量。层级越高越稳定、越难推翻。后续开发遇到两条原则给出相反方向时，查层级就能裁决。

内容分两类，两类之间有推导关系：

### First Principles（第一性原理）

对项目最本质的洞察——理解了这一条，就能推导出下游的多个设计决策。不是表面的事实罗列，而是对"这个项目为什么必须是这样"的最深层解释。

一条第一性原理如何锁定整个系统的设计方向：

Unix 的设计者面对的问题是：操作系统需要处理磁盘、终端、打印机、网络、进程间通信等完全不同的 I/O 对象。Thompson 和 Ritchie 的洞察：**"Everything is a file."** 从这一条推出：管道能工作（进程输出就是文件流）、设备用同一套 API 访问（/dev/ 就是文件）、socket 可读可写（文件描述符）、权限模型统一（所有对象共享 read/write/execute 语义）。一条洞察，整个 I/O 架构的形状确定了。它不是"I/O 系统要设计好"（口号），也不是"我们需要统一的 I/O 接口"（方案）——只有先看到"所有 I/O 对象的本质是同一种东西"，统一接口才从一个方案变成一个必然。

另一个领域：SQLite 面对的问题是嵌入式场景中如何存储结构化数据。D. Richard Hipp 的洞察：**"SQLite is not designed to replace Oracle. It is designed to replace fopen()."** 从这一条推出：零配置无服务器（fopen() 不需要 DBA）、单文件存储（fopen() 操作的就是一个文件）、可嵌入任何程序（fopen() 在所有平台都存在）、不追求并发吞吐量（fopen() 也不追求）。这条洞察不只决定了 SQLite 该有什么——更重要的是决定了它不该有什么。它不是"做一个轻量级数据库"——"轻量级"是结论，不是洞察。洞察是看到了嵌入式数据存储的本质需求和 fopen() 的本质局限之间的精确缝隙。

第一性原理不是一句金句。核心洞察可能一句话就能说清，但它需要被充分展开——从多个角度论证为什么这是真的，推导它意味着什么和不意味着什么，标注什么情况下它会失效。"Everything is a file" 是种子；把它展开成"为什么所有 I/O 对象本质上是同一种东西、这意味着接口必须统一、不意味着所有操作都该同步、如果出现无法映射为文件语义的对象则需要重新审视"——这才是一条能真正支撑后续决策的第一性原理。目的是让读者不只知道结论，还理解为什么这个结论是不可避免的，从而能在遇到新情况时自主判断。

以上示例都是从零构建的系统。重构项目的第一性原理有一个额外的来源：对现有系统的诊断。一个 1500 行的 Controller 同时处理路由、认证和数据访问——这是症状。"系统从未把模块边界当作硬约束"是一种根因；"架构约束缺乏机械性守护机制"是另一种。同一个症状，不同的根因分析锁定不同的重构方向。诊断的目的不是列一张技术债清单，而是从症状追溯到根因——根因就是第一性原理的种子。

检验三问：
1. 去掉"我们认为""我们决定"，还成立吗？不成立 → 不是洞察，是偏好
2. 从这一条能推导出多个下游决策吗？只能推出一个 → 可能还不够深，下面还有更本质的洞察
3. 如果这条不成立，项目的设计根基是否需要重来？不需要 → 不是第一性原理，是普通约束

警惕伪装成第一性原理的偏好。用户说"必须用 Python"——追问：是因为团队只有 Python 经验（人力约束，可以是第一性原理的一部分），还是觉得 Python 好用（偏好，可以挑战）？

### Heuristics（启发式）

从 First Principles 自然推导出的判断规则——后续开发中遇到岔路时的导航依据。形式不限，以下只是常见的几种，不是完整分类：取舍规则（"当 X 和 Y 冲突时，默认选 X"）、架构模式（"架构与策略分离"）、优先级排序（"A > B > C"）、方向性隐喻（"不造大脑，造手术室"）、禁区（"永远不做 X"）。

关键是两点：每条能追溯到一条 First Principle，并且标注了适用边界——什么情况下这条规则失效，需要停下来重新评估。知道边界比知道规则本身更重要。

延续前面的例子：Unix 的 "Everything is a file" 自然产生了 Heuristics——"新的 I/O 对象优先映射为文件语义，只在文件语义确实无法表达时才引入新抽象"、"权限模型统一用 rwx，不为特殊对象发明专用权限体系"。边界：当某种对象的核心操作不是读写（如 GPU 计算），文件抽象开始失灵，需要重新评估。SQLite 的 "replace fopen()" 产生了——"拒绝所有需要 DBA 的功能请求"、"单文件是硬约束，任何需要多文件协调的方案直接否决"。边界：当并发写入成为真实瓶颈时，说明场景已经超出了 fopen() 的定位，应该换工具而不是改 SQLite。

追溯不到 First Principle 的 Heuristic → 要么有隐含的第一性原理没挖出来，要么是无根的偏好。

## 工作流

### 1. 了解项目

查看 `.nn/materials/` 有无用户提供的参考资料，如有先读一遍。扫描项目目录，建立对项目当前状态的理解。

如果项目目录基本是空的，理解主要来自后续对话。如果有实质性的代码，代码本身就是最大的信息源——需要一次系统性的诊断——不是扫一眼目录结构，而是深入阅读核心模块的代码。目标不是列一张技术债清单，而是理解这个系统在时间压力下发生了什么——哪些设计经受住了考验，哪些在压力下变形了，变形的根因是什么。每一个设计决策背后可能藏着深层洞察，也可能只是历史惯性——需要和用户一起区分。"代码中选择了事件溯源架构——是因为每次状态变更都有法律后果，还是当时团队里有人推崇这个模式？"前者是第一性原理的素材，后者是可以推翻的惯性。

诊断的思维方式是从可观察的症状追溯到结构性根因。老代码库的问题千差万别，但退化模式高度趋同——

**结构层面**——模块边界是否还清晰，还是职责已经互相渗透？依赖关系是单向的还是循环的？有没有"什么都管"的巨型模块，改任何功能都要动它？全局可变状态是否泛滥，函数的行为取决于之前所有函数的执行历史而非输入参数？结构腐化的根因通常是系统从未把模块边界和状态所有权当作硬约束——每次加功能时选了成本最低的路径（直接往现有模块上堆代码），短期节省的时间以复利变成了长期负债。

**契约层面**——系统的行为依赖多少未被代码显式表达的"潜规则"？函数调用顺序依赖（B 必须在 A 之后调用，因为 A 初始化了某个全局状态）、魔法值（`if status == 3`，3 代表什么没有任何定义）、字典冒充结构体在多层函数间传递（每个函数默默假设字典里有某些键，一旦上游少传一个字段，报错在下游十层之后）。契约隐式化的根因通常是接口设计从未区分公共合约和内部实现——所有东西对所有人可见，任何人都可以依赖任何内部细节。

**反馈层面**——测试覆盖到什么程度？是充分的、只覆盖 happy path、还是几乎空白？错误处理是统一设计过的，还是走向两个极端——要么完全不处理，要么 catch-all 后静默继续，系统在错误状态下运行直到很久以后才被发现？代码的可测试性如何——能否独立测试一个模块，还是函数内部直接实例化依赖、调用网络、依赖全局状态？反馈断裂的根因通常是可测试性和错误可观测性从未被当作设计约束。

**演进层面**——依赖是否健康（版本严重落后？已废弃的库仍在使用？）？有没有早期技术选型变成了沉重的锁定——当初合理的选择在需求和规模变化后变成了约束，但替换成本随时间呈指数增长？构建和部署是否脆弱——必须按特定顺序执行手动步骤才能成功？演进僵化的根因通常是系统从未为"变化"本身做过设计。

不需要按这四个维度逐一扫描——它们是诊断的视角，不是检查清单。阅读代码时带着这些视角，自然会注意到问题的模式。关注那些影响最广、出现最频繁的模式——代码中反复出现的结构性问题，就是系统缺失的第一性原理的负像。

同时注意正面信号：哪些模块设计得好？哪些抽象经受住了时间考验？好的设计同样是第一性原理的素材——它们揭示了系统中哪些原则被隐式地遵守了，值得显式化和推广。

### 2. 与用户对话

默认立场：用户说的每一句话都需要验证。人在描述自己想要什么时天然不可靠——会把别处见过的特性当成自己的需求，会高估自己对领域的了解，会用笼统的词掩盖还没想清楚的部分，会自信地说出完全错误的领域知识。这不是例外，是常态。

Stop guessing, start asking. 不猜测，问。不接受，验证。不填补空白，暴露空白，避免一切可能由于当前项目的洞察不够而导致的幻觉。写进 first-principles.md 的每一条洞察必须经过对话中的明确验证——推断出来的东西看起来合理，但合理不等于正确，写进第一性原理的错误假设会传导到项目的每一个角落。

"我需要实时同步"——真的需要？追问具体场景：没有实时同步会发生什么不可接受的后果？说不出来，这就不是第一性原理。不确定的事情更要反复问——不确定往往意味着触及了真实世界的具体约束（领域经验、行业规范、物理限制），这些正是第一性原理的原材料。不确定不是搁置的理由，是继续追问的信号。

目标是触及项目最本质的洞察，然后从中推导出可执行的判断规则。

挖掘 First Principles 的方向——这个项目存在的根本原因是什么？这个领域里什么是不可改变的物理约束？系统面对的核心矛盾是什么（什么和什么在根本上互相对抗）？这个领域的反馈信号是什么——编译器、用户行为数据、人工审核、还是没有客观裁判？反馈信号的性质决定了自动化验证循环能否收敛。系统如何随时间退化也是第一性原理的领域——即使是全新项目，这个维度也不可跳过。追问：这些原则在经历几十轮开发后会以什么方式被侵蚀？模块边界会在赶工期中被突破吗？接口契约会随着需求变更悄悄变成隐式的吗？测试会从设计约束退化为事后补充吗？好的第一性原理不只定义系统该是什么样，还要预见系统会怎么死——如果一条原则没有内置防腐蚀机制，它就只是写在纸上的愿望。

挖掘 Heuristics 的方向——当两个都合理的方案摆在面前，用户倾向哪个？速度 vs. 质量？简单 vs. 灵活？什么是明确不做的（边界）？怎样算"做完了"——质量属性的优先级排序？

表面需求下面藏着第一性原理。用户说"我要做一个 X"——X 解决什么问题？那个问题为什么存在？什么根本性的矛盾或约束造成了它？继续追问直到碰到不可再分解的洞察。

已有代码库的对话有一个完全不同的开场：不是"你想建什么"，而是"哪里最痛"。用户说"这个系统需要重构"——追问：什么事件触发了这个决定？是一次生产事故？是新功能开发速度慢到不可接受？是关键人员离职后没人敢碰代码？触发事件往往直接指向系统最缺失的第一性原理。

重构的第一性原理通常不只关于"系统该是什么样"，更关于变换本身——什么属性是当前系统缺失但重构后必须具备的？变换的边界在哪里——什么必须保留（外部接口、数据格式、用户可见行为）、什么可以改变？变换的约束是什么——能否停机重写还是必须渐进式迁移？

追问方向：这个系统曾经试图成为什么？哪些部分做到了、哪些偏离了？偏离是因为原始设计有缺陷，还是因为需求变化超出了设计的弹性？用户对当前代码的哪些部分有信心，哪些部分有恐惧（"碰这里就出 bug"）？恐惧区域往往是第一性原理缺失最严重的地方。结合第一步的代码诊断发现，把技术症状和用户感受对照——两者指向同一个问题时，那就是最核心的第一性原理候选。

对话节奏：一次一个方向，听完再展开。回答模糊时推回去——"性能要好"不是 First Principle 也不是 Heuristic，追问到具体。注意用户说的和用户想的之间的差距——用户说"简单的 TODO 应用"但描述了实时协作和离线支持，指出矛盾。

遇到用户能感觉到某件事重要但说不清为什么——不要替用户编一个理由，把可能的洞察视角摆出来让用户识别。同一个问题，不同的哲学视角会推导出完全不同的系统形态。用户想做一个内容平台——核心矛盾是什么？如果是"创作者和消费者的注意力是零和博弈"，系统会走向算法推荐和流量分配；如果是"好内容的发现成本太高"，系统会走向策展和搜索；如果是"创作的门槛太高导致供给不足"，系统会走向创作工具和模板。三个视角都言之成理，但推导出的架构、功能优先级、甚至商业模式截然不同。在这个层次上没有对错，只有视角——而视角服务于用户真正想解决的问题。用户不需要自己发明这些视角，只需要识别哪个最接近自己的处境。

当你能预测用户对下一个问题的回答时，对话接近完成。

### 3. 起草与验证

将对话中提取的内容写入 `.nn/Memory/A-principles/first-principles.md`。组织为层级结构：哪条洞察是最根本的（整个项目的存在理由）？哪些从它推导出来？哪些在前两者约束下展开？确保每条 Heuristic 能追溯到对应的 First Principle，冲突时能查到谁服从谁。

每完成一版，spawn orient-auditor 子智能体做隔离审视（独立审计 first-principles.md 的洞察深度、权力结构和推导链完整性）。调用提示词："审计 `.nn/Memory/A-principles/first-principles.md`。先扫描项目目录和 materials/ 建立独立理解，重点发现主对话可能遗漏的方向性盲区，同时检查洞察深度、权力结构、根基测试、Heuristic 质量。返回盲区发现和关键质量问题。"审计发现问题则修改，修改后重新 spawn，直到通过。

### 4. 和用户确认

把最终文件和用户过一遍。这是确认项目方向的最后环节。
