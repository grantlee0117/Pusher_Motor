# 开发札记

本目录用于记录开发过程中的心得、踩坑、排查思路和阶段性判断。

这里的内容偏向“开发过程记录”，不等同于最终规范或唯一事实来源。若某条经验已经稳定下来，并且会影响团队协作、编译流程、硬件接口或调试方式，应同步沉淀到对应正式文档中。

## 适合放在这里的内容

- 某个问题的排查过程和结论。
- 某次硬件联调、换板、外设验证的记录。
- 编译、下载、调试工具链中的经验和注意事项。
- 对仓库结构、模块边界、后续重构方向的阶段性想法。
- 尚未成熟到可以写入正式规范，但值得后续团队参考的背景信息。

## 不适合放在这里的内容

- 当前团队协作约定：应写在根目录 `README.md`。
- 固件编译流程：应写在 `../Build.md`。
- 串口命令和调试协议：应写在 `../串口调试命令.md`。
- 硬件资料索引和板级资料说明：应写在 `../../hardware/README.md`。
- 已确认过时的历史文档：应放入 `../deprecated/`。

## 建议写法

- 一件事情尽量单独写成一个文件，避免把所有记录堆在一个大文档里。
- 文件名使用英文小写和连字符，例如 `hardware-bringup.md`、`build-toolchain-notes.md`。
- 文档开头写清楚记录对象、时间背景和当前结论。
- 如果记录来自代码、原理图、测试现象或串口日志，尽量给出对应路径或事实来源。
- 如果某条记录后续已经转为正式约定，在原文中补一句说明并链接到正式文档。

## 可选分类

后续内容多起来之后，可以按场景逐步增加子目录或文件，例如：

```text
docs/dev-notes/
├── README.md
├── build-toolchain-notes.md
├── hardware-bringup.md
├── debugging.md
└── repo-maintenance.md
```
